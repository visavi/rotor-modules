# Версионирование

Правила совместимости ядра и модулей. Действуют начиная с ядра 14.1.0.

## Обещание ядра

Ядро версионируется по [semver](https://semver.org/lang/ru/): `мажор.минор.патч`.

- **Патч** (14.1.0 → 14.1.1) — исправления. Модули не затрагиваются.
- **Минор** (14.1 → 14.2) — новые возможности. Публичный API ядра не ломается, модули продолжают работать без обновления.
- **Мажор** (14.x → 15.0) — допустимы ломающие изменения API. Модули, собранные под прошлый мажор, могут перестать работать.

Каждое ломающее изменение сопровождается **мостом совместимости** — прослойкой, которая держит старое поведение в течение одного мажора. Мост помечается `@deprecated` и удаляется в следующем мажоре.

> Историческое исключение: переход хранения дат с timestamp на datetime вошёл в 14.1.0 — до принятия этих правил. Для него действует мост (`SITETIME`, `dateFixed()` с int/string), который будет удалён в 15.0.

## Что считается публичным API

На это модули могут опираться — внутри мажора оно стабильно:

- хуки (`hooks.php`, `@hook` в шаблонах);
- глобальные хелперы из `app/helpers.php`;
- структура `module.php` и жизненный цикл модуля (установка, миграции, публикация);
- базовые модели ядра (`User`, `Comment`, `File`, ...) — имена таблиц, полей и связей;
- трейты интеграции (`FeedableTrait`, `SearchableTrait`, `HandlesComments`, ...).

Всё остальное — внутренности ядра: может измениться в любом миноре без предупреждения. Если модулю не хватает публичного API — лучше попросить хук или хелпер в ядре, чем лезть внутрь.

## Что обязан декларировать модуль

В `module.php`:

```php
'version'  => '1.2.0',   // версия модуля, semver
'requires' => '14.1.0',  // минимальная версия ядра, под которую модуль собран
```

Семантика `requires` — «собран и проверен под это ядро»:

- при установке и обновлении модуля ядро откажет, если его версия ниже `requires`;
- благодаря обещанию ядра модуль с `requires: 14.1.0` работает на всём мажоре 14.x начиная с 14.1.

## Когда бампить requires

- Модуль начал использовать API, появившееся в новом ядре → поднять `requires` до этой версии.
- Вышел новый мажор ядра, модуль проверен на нём → поднять `requires` до нового мажора и выпустить версию модуля.
- Просто вышел новый минор/патч ядра → **ничего делать не нужно**.

## Чек-лист автора модуля

1. Использовать только публичный API (список выше).
2. Держать `requires` честным: не выше, чем реально нужно, не ниже, чем реально проверено.
3. Вести `changelog.md` — секция `## <версия>` попадает в реестр и видна при обновлении.
4. Перед мажором ядра: прогнать модуль на нём, поправить использование удалённых мостов, выпустить версию с новым `requires`.

## Что видит владелец сайта

- Старый модуль на новом ядре того же мажора — работает.
- После смены мажора — модуль работает, пока жив мост; в админке виден баннер о доступных обновлениях модулей.
- Модуль, чей автор не обновил его к новому мажору, после удаления моста перестанет работать — это ожидаемое завершение поддержки, а не ошибка ядра.
